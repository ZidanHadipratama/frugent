**Software Requirements Specification**
Version: 2.0
Status: Draft
Author: Mochamad Zidan Hadipratama

## 1. System Overview

Frugent is a **file-based multi-agent coordination system**. It has no runtime, no server, and no UI. It is entirely composed of:

- **Skill files** — markdown instructions installed globally per agent tool
- **Document templates** — markdown files that agents read and write during a project
- **A tracker script** — a Python script agents invoke to check quota usage
- **Conventions** — agreed naming, update triggers, and escalation rules

It is installed once globally and used across any project. Agents scaffold project docs when the developer asks.

## 2. System Architecture

```
~/.claude/CLAUDE.md          ← Claude Code skill (global)
~/.gemini/GEMINI.md          ← Gemini CLI skill (global)
~/.frugent/
├── tracker.py               ← Usage tracker script (agent-invoked)
├── templates/               ← Document templates
│   ├── plan.md
│   ├── contracts.md
│   ├── briefing.md
│   ├── log.md
│   ├── test-cases.md
│   ├── qa-report.md
│   └── codebase-analysis.md ← Generated by Gemini (deep init only)
└── usage.json               ← Usage tracker data

your-project/
├── docs/
│   ├── plan.md              ← Phases, tasks, assignments
│   ├── contracts.md         ← Interfaces between components
│   ├── briefing.md          ← Session bootstrap (updated per session)
│   ├── log.md               ← Progress + blockers + handoffs + suggestions
│   ├── test-cases.md        ← What to test per feature
│   ├── qa-report.md         ← Test results
│   └── codebase-analysis.md ← Codebase scan (deep init only)
```

## 3. Skill File Requirements

### 3.1 Global Planner Skill (CLAUDE.md)

**Installed at:** `~/.claude/CLAUDE.md`
**Read by:** Claude Code on every session start
**Max length:** 200 lines

Must instruct Claude Code to:
- Always identify its current role at session start (planner / executor)
- Read `docs/briefing.md` before doing anything else
- Run `python ~/.frugent/tracker.py status` at session start and report budget
- Produce structured outputs in the correct document format
- Never implement features outside its assigned task
- Raise blockers immediately in `log.md` rather than hallucinating a solution
- Write a handoff entry in `log.md` when approaching context/quota limits
- Log out-of-scope ideas to `log.md` with a `[suggestion]` tag instead of implementing them
- When planning: specify intent and contracts only — function names, input/output types, business logic steps, libraries to use, and edge cases to handle. Never write pseudocode or implementation-level detail.
- When asked to init: offer three modes (quick / deep / skip), check for existing PRD/SRS, and scaffold accordingly
- In deep init mode: instruct the developer to run Gemini for codebase analysis, then read `docs/codebase-analysis.md` to produce the plan

### 3.2 Global Executor Skill (GEMINI.md)

**Installed at:** `~/.gemini/GEMINI.md`
**Read by:** Gemini CLI on every session start
**Max length:** 200 lines

Must instruct Gemini CLI to:
- Identify itself as a free-tier executor
- Read `docs/briefing.md` before starting
- Run `python ~/.frugent/tracker.py status` at session start and report budget
- Append progress entries to `log.md` after every task
- Append blocker entries to `log.md` immediately when stuck
- Flag complex architectural decisions to Claude Code rather than deciding alone
- Write a handoff entry in `log.md` before quota runs out
- When asked to analyze a codebase: read the entire project and fill in `docs/codebase-analysis.md`

## 4. Document Specifications

### 4.1 plan.md
**Written by:** Planner (Claude Code)
**Updated:** Per phase

Required sections:
- Project summary (1 paragraph)
- Phase list with goals per phase
- Per phase: features, tasks per feature, complexity tag (`standard` / `complex`)
- Dependency map between features (what must be done before what)
- Agent assignment per task (`standard` → Gemini, `complex` → Claude)

### 4.2 contracts.md
**Written by:** Planner
**Updated:** Per feature

Required sections per interface:
- Name and purpose
- Request/input shape
- Response/output shape
- Error cases
- Which agent owns this component

### 4.3 briefing.md
**Written by:** Planner (or updated by any agent at session end)
**Updated:** Per session start

Required sections:
- Current phase and feature
- Last completed task (from log.md)
- Current task and assignment
- Known blockers (from log.md)
- Files the current agent owns
- Files the current agent must not touch
- Quota budget (from `python ~/.frugent/tracker.py status`)

### 4.4 log.md
**Written by:** Any agent
**Updated:** Continuously

A single append-only log with tagged entries. Each entry has:
- Timestamp
- Agent name
- Entry type tag: `[progress]`, `[blocker]`, `[handoff]`, `[suggestion]`, `[retro]`

**[progress] entries:**
- Task completed
- Files modified
- Next task

**[blocker] entries:**
- Task being attempted
- Nature of blocker (ambiguity / technical / dependency / environment)
- What is needed to unblock
- Severity (blocking / degraded / minor)

**[handoff] entries:**
- Session summary
- Completed in this session
- In progress (exact state)
- Remaining tasks
- Files modified this session
- Instructions for next agent/session to resume

**[suggestion] entries:**
- Suggestion description
- Why it would be valuable
- Estimated complexity

**[retro] entries:**
- Phase number and name
- What went well
- What went wrong
- Changes to make next phase

### 4.5 test-cases.md
**Written by:** Planner
**Updated:** Per feature

Required fields per test case:
- Test ID
- Feature being tested
- Input
- Expected output
- Edge cases
- Pass criteria

### 4.6 qa-report.md
**Written by:** QA agent or developer
**Updated:** Per feature tested

Required fields per test case:
- Test ID (from test-cases.md)
- Result (pass / fail / blocked)
- Actual output
- Notes
- Severity if failed (critical / major / minor)

### 4.7 codebase-analysis.md
**Written by:** Gemini CLI (deep init) or Claude Code (quick init)
**Updated:** Once during project init

Required sections:
- Tech stack (languages, frameworks, database, tools)
- Project structure (directory tree of key folders)
- Modules & components (what exists, purpose, status: built/partial/broken)
- Existing interfaces (APIs, function contracts, component props)
- Existing tests (framework, location, coverage)
- Dependencies (installed packages and their purpose)
- Git state (current branch, open branches, uncommitted work)
- Current state summary (what's done, in progress, broken, or missing)

## 5. Agent Role Definitions

| Role | Tool | Cost Tier | Owns |
|---|---|---|---|
| Planner | Claude Code | Paid | plan, contracts, test-cases, briefing |
| Executor | Gemini CLI | Free | Implementation within contracts |
| Executor (complex) | Claude Code | Paid | Complex implementation, AI logic |
| Developer (Human) | — | — | ENV vars, final approval, unblocking |

## 6. Cost Routing Rules

Every task in `plan.md` is tagged with a complexity level:

| Complexity | Routed to | Rationale |
|---|---|---|
| `standard` | Gemini CLI (free) | Boilerplate, CRUD, UI, repetitive tasks |
| `complex` | Claude Code (paid) | Architecture, AI logic, ambiguous requirements |

**Rules enforced by skill files:**

1. **Gemini first** — default all tasks to `standard` unless clearly complex
2. **Claude for complexity** — architecture decisions, AI logic, cross-cutting integration
3. **Escalate don't guess** — if Gemini encounters something architecturally significant, it raises a `[blocker]` in `log.md` rather than deciding alone
4. **Handoff before exhaustion** — any agent approaching limits writes a `[handoff]` entry in `log.md`
5. **Intent vs implementation** — Claude specifies what and why, Gemini decides how. If a plan reads like pseudocode, it's too detailed.

## 7. Git Conventions

- Work on `main` or a single `dev` branch by default
- Feature branches (`feature/[name]`) only when working on parallel features
- Commit message format: `type(scope): description` — e.g. `feat(auth): add JWT login endpoint`
- No integrator role — developer merges branches when needed

---

## 8. Definition of Done

A task is done when:
1. Code is written and works locally
2. `[progress]` entry appended to `log.md`
3. Developer or QA has confirmed it works as intended
4. No open `[blocker]` entries related to this task

A feature is done when:
1. All tasks are done
2. `qa-report.md` shows all test cases passing
3. `briefing.md` is updated for the next feature
4. Developer has confirmed

## 9. Usage Tracker — `tracker.py`

### 9.1 Purpose

A Python script at `~/.frugent/tracker.py` that tracks active usage across Claude Code and Gemini CLI. Agents invoke it via shell command at session start. The developer never needs to run it separately.

### 9.2 Claude Code Tracking

**Data source:** `~/.claude/projects/[project-hash]/[session-id].jsonl`

Claude Code writes every prompt and response to local JSONL files with timestamps. The tracker reads these to reconstruct active query-to-response intervals, excluding idle time.

**Tracking logic:**
```
For each session JSONL file:
  For each user message entry:
    interval_start = user message timestamp
    interval_end   = next assistant message timestamp
    active_time   += (interval_end - interval_start)

Weekly total = sum of active_time across all sessions this week
```

**Thresholds:**
- 5-hour rolling window: warn at 4 hours active
- 40-hour weekly cap: warn at 35 hours

### 9.3 Gemini CLI Tracking

**Primary data source (automatic):** `~/.frugent/gemini-telemetry.jsonl`

`setup.sh` configures Gemini CLI's built-in OpenTelemetry exporter to write telemetry data (including per-model token counts) to a local JSONL file. The tracker reads this file automatically — no wrapper script or manual recording needed.

**Configuration added to `~/.gemini/settings.json`:**
```json
{
  "telemetry": {
    "enabled": true,
    "target": "local",
    "outfile": "~/.frugent/gemini-telemetry.jsonl",
    "logPrompts": false
  }
}
```

**Fallback data source (manual):** `tracker.py record-gemini '<json>'`

If telemetry is unavailable (e.g. user disabled it, format changed), the developer can manually record usage by passing Gemini's `--output-format json` output to the tracker. The GEMINI.md skill also instructs Gemini to report `/stats model` output in log.md entries as a last resort.

**Token tracking:** Pro and Flash tokens are tracked separately. Only Pro tokens count against the daily budget.

**Thresholds:**
- Daily Pro tokens: warn at ~25,000 tokens (~80% of observed daily budget)
- Flash tokens: no warning

### 9.4 Data Storage

**File:** `~/.frugent/usage.json`

```json
{
  "week_start": "2026-03-02",
  "claude": {
    "weekly_total_mins": 210,
    "sessions": [
      {
        "date": "2026-03-06",
        "project": "frugent",
        "active_mins": 87,
        "intervals": 23
      }
    ]
  },
  "gemini": {
    "daily_totals": [
      {
        "date": "2026-03-06",
        "pro_tokens": 18420,
        "flash_tokens": 94200,
        "prompts": 34
      }
    ]
  }
}
```

### 9.5 Invocation

Agents run this command at session start (instructed by skill files):
```bash
python ~/.frugent/tracker.py status
```

Output is reported inline in the agent session. No separate terminal needed.

### 9.6 Warning Output

```
CLAUDE CODE
  5-hour window: 4h 12m of ~5h (84%) — prepare handoff soon
  Weekly: 32h 15m of ~40h (81%) — 7h 45m remaining

GEMINI CLI
  Pro tokens today: 18,420 of ~30,000 (61%) — OK
  Flash tokens: 94,200 (no limit concern)
  Source: telemetry
```

### 9.7 Auto-Reset

Weekly counters reset automatically when the tracker detects a new week (Monday). Daily Gemini counters reset when a new day is detected. No manual reset command needed.

### 9.8 Graceful Degradation

If Claude's JSONL format changes or files are unreadable, the tracker outputs:
```
CLAUDE CODE
  Unable to read usage data — JSONL format may have changed
  Proceed with caution and monitor quota manually
```

Same for Gemini if JSON output format changes. The tracker never crashes or gives incorrect numbers — it either reports accurately or says it can't.

### 9.9 Constraints

- Reads Claude JSONL files as read-only — never modifies Claude's logs
- Gemini telemetry requires `setup.sh` to configure `~/.gemini/settings.json` — done automatically during install
- Gemini telemetry sets `logPrompts: false` to avoid logging sensitive prompt content
- If telemetry is disabled by user, falls back to manual `record-gemini` command
- Claude active time is an approximation
- Gemini Pro daily budget (~30,000 tokens) is an observed estimate
- Requires Python 3.6+ standard library only — no pip installs

## 10. Installation

### 10.1 What Gets Installed

```
~/.frugent/
├── tracker.py               ← Usage tracker script
├── templates/               ← Document templates (7 files)
├── usage.json               ← Created on first run
└── gemini-telemetry.jsonl   ← Written by Gemini CLI (auto)

~/.claude/CLAUDE.md          ← Planner skill file
~/.gemini/GEMINI.md          ← Executor skill file
~/.gemini/settings.json      ← Updated with telemetry config
```

### 10.2 How to Install

A single `setup.sh` script that:
1. Creates `~/.frugent/` and copies `tracker.py` and `templates/`
2. Copies `CLAUDE.md` to `~/.claude/` (creates dir if needed)
3. Copies `GEMINI.md` to `~/.gemini/` (creates dir if needed)
4. Configures Gemini CLI telemetry in `~/.gemini/settings.json` (merges with existing settings, backs up if needed)

```bash
git clone <repo> /tmp/frugent-install
bash /tmp/frugent-install/setup.sh
```

Or manually:
```bash
cp -r frugent-dist/.frugent ~/
cp frugent-dist/CLAUDE.md ~/.claude/CLAUDE.md
cp frugent-dist/GEMINI.md ~/.gemini/GEMINI.md
```

### 10.3 New Project Setup

No CLI command needed. The developer tells Claude Code:
> "Init frugent for this project"

Claude reads CLAUDE.md, which instructs it to copy templates from `~/.frugent/templates/` into `./docs/`. Done in the same terminal, same session.

---

## 11. Constraints

- Skill files must stay under 200 lines each
- All documents must be plain markdown — no special syntax or tooling
- The system must work without any internet access beyond what the agent tools themselves require
- No secrets or API keys are ever stored in any Frugent document
- Usage tracker requires Python 3.6+ (standard library only)
- No external package managers (npm, pip) required for installation
